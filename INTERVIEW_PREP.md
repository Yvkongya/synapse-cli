# 第一部分：Synapse CLI — 自研 Agent CLI

## 1.1 核心架构篇

### Q1：ReAct 循环和 Function Calling API 的交互流程，请完整说一遍。

**考察点**：Agent 执行全链路理解

**回答框架**（建议按顺序）：

```
1. 构造 tools 参数
   - 每个工具定义为一个 JSON Schema（name / description / parameters）
   - ToolRegistry 收集所有 ToolDef，`registry.defs()` 返回 List
   - LLM API 的请求体中放入 tools 参数

2. 发送给 LLM
   - LlmClient.call(history, tools)
   - 历史消息包含 system prompt（含记忆）+ 对话轮次
   - tools 以 function calling 格式传入

3. 解析响应
   - 检查 resp.toolCalls：是结构化 JSON，不是文本
   - 每个 ToolCallMsg 包含：id、function.name、function.arguments
   - 100% 解析成功（和文本 TOOL 前缀方案对比）

4. 调度工具
   - 串行执行每个 ToolRunner
   - 经历权限检查（Auto / HITL）→ 参数解析 → 工具执行
   - 结果通过 addToolResult 写回 history

5. 继续循环
   - 把 tool result 加回 history
   - 下一轮调 LLM 时带上
   - LLM 判断是否还需要调用工具，或直接返回最终答案
```

**加分点**：主动提之前的方案演进（文本 TOOL 前缀 → Function Calling），展示你做过多次迭代。

### Q3：为什么工具执行从并行改成了串行？

**考察点**：并发安全意识、踩坑复盘

**背景**（面试官知道会加分）：最初和 README 一样写的是 `CompletableFuture.runAsync` 并行执行，上线测试两个工具同时跑时出了问题。

**两个具体问题**：

| 问题 | 现象 | 根因 |
|------|------|------|
| Scanner 抢读 | 两个工具同时打印审批提示，用户输入一次却有两个线程在等 nextLine() | Scanner 不是线程安全的，多线程读 stdin 行为未定义 |
| history 并发写 | 两个工具结果同时写入 ArrayList，导致数据错乱或下标越界 | ArrayList 不是线程安全的，add() 在并发下会丢数据或抛异常 |

**回答要点**：

> "最初用了 CompletableFuture 并行执行，但遇到两个具体问题：第一，多个工具同时打印审批提示框，两个线程同时读 Scanner，用户的输入不知道被哪个线程消费了，另一个线程还在死等；第二，每个工具执行完都要往 history 列表加一条 tool result，ArrayList 的 add 不是原子操作，两个线程同时写会出现数据丢失。改成串行后这两个问题都解决了，代价是工具执行时间变成累加不是取 Max，但对于 CLI 场景用户感知不明显。"

**如果面试官追问"那性能怎么优化"**：

> "如果以后需要并行，我会用显式的线程池 + 一个线程安全的队列收集结果 + 所有结果收齐后再统一写回 history，这样既并行又安全。"

### Q4：上下文压缩策略怎么设计的？为什么 tool 消息也要保留？

**考察点**：上下文窗口管理

**回答要点**：

- 触发条件：历史超过 20 条
- 保留策略：
  - system prompt（含记忆）完整保留
  - 中间部分（第 1 到倒数第 10 条）调 LLM 压缩
  - 最近 10 条原始保留
- 压缩时保留的内容：
  - user 消息：用户意图
  - assistant 内容回复：模型的推理过程
  - **tool 消息**：工具执行结果 ← 这是关键

**为什么 tool 消息不能丢**：

> "tool 消息里包含的是工具执行的返回数据，比如文件内容、搜索结果、命令输出。如果压缩时只保留 user/assistant，下一轮 LLM 就看不到前面工具执行的结果了，会重复调用同一个工具或者凭空编造结果。这是 Agent 系统和普通聊天系统最大的区别——工具执行上下文是连续决策的依据。"

**考察点**：安全意识、兜底设计

**回答框架**（三步走）：

```
第三步：指出方向
  "如果要更可靠，可以加规则引擎做第一道过滤——已知危险操作直接拦截，剩下的交给 LLM 判断。"
```

### Q6：三级权限为什么这样分？file_write Level 1 而 bash Level 2？

**考察点**：风险分级的实际考量

**回答要点**：

| 级别 | 工具 | 理由 |
|------|------|------|
| Level 0 自动 | file_read / web_search / ask_user | 只读操作，没有副作用，不会造成数据损失 |
| Level 1 可跳过 | file_write | 写文件有风险但可逆（能删/能回滚），且用户大概率就是想让 Agent 写文件 |
| Level 2 必须确认 | bash | 命令执行破坏性不可控，用户按 y/N 必须显式输入 yes |

**加分点**：主动提及参考了 Claude Code 的 permission model（自动允许 / 首次确认 / 每次确认 / 极危险）。

## 1.3 安全边界篇

### Q7：BashTool 怎么防止 shell 注入？

**考察点**：安全工程意识（高频考点）

**三层防御**：

| 层级 | 措施 | 解决的问题 |
|------|------|-----------|
| 命令名白名单 | 提取第一个 token 精确匹配 `SAFE_COMMANDS` Set | 防止 LLM 编造命令名 |
| shell 操作符拦截 | 检查参数中是否包含 `;` `|` `&&` `` ` `` `$()` 等 | 防止命令拼接绕过 |
| ProcessBuilder | 代替 `Runtime.getRuntime().exec()` | 避免 shell 解释参数 |

**关键区别**：

```java
// ❌ 危险：被 shell 解释
Runtime.getRuntime().exec("cat file.txt; rm -rf /");
// 实际上会执行 cat → 然后执行 rm

// ✅ 安全：参数作为独立数组传递，不被 shell 解释
new ProcessBuilder("cat", "file.txt; rm -rf /")
// 实际上试图找一个名为 "cat" 参数为 "file.txt; rm -rf /" 的命令
```

### Q8：文件读写的工作目录检查怎么实现的？

**考察点**：路径穿越防御

**回答要点**：

```java
Path target = Paths.get(path).normalize().toAbsolutePath();
Path cwd = Paths.get(".").toAbsolutePath().normalize();
```

## 1.4 MCP 协议篇

### Q9：MCP 协议的消息格式？Content-Length 为什么不能只 read 一次？

**考察点**：协议帧解析、网络编程基础

**回答要点**：

消息格式：
```
Content-Length: 123\r\n\r\n{json body}
```

**BufferedReader.read(buf, off, len) 不保证一次读满**：

> "Java 的 Reader.read(char[], off, len) 的 Javadoc 明确说了：'尝试读取最多 len 个字符，但可能读取更少'。网络流不是文件流，数据是分块到达的。如果一条 Content-Length: 4096 的消息，第一次 read 可能只返回了 1024 字节，剩下的还在缓冲区里。所以我用循环读直到 totalRead == len。"

**代码**：
```java
char[] buf = new char[len];
int totalRead = 0;
while (totalRead < len) {
    int n = reader.read(buf, totalRead, len - totalRead);
    if (n == -1) throw new IOException("流提前关闭");
    totalRead += n;
}
```

### Q10：MCP 连接的生命周期？

1. 启动外部进程（ProcessBuilder）
2. 发送 `initialize` 请求（握手，含协议版本）
3. 接收 `initialize` 响应（确认版本兼容）
4. 发送 `notifications/initialized` 通知
5. 发送 `tools/list` 获取工具列表
6. 按需发送 `tools/call` 调用工具
7. 关闭时 `process.destroy()`

## 2.1 RAG 全流程篇

### Q12：完整走一遍 RAG 处理流程

**考察点**：RAG 全链路理解（核心必问题）

**回答框架**（按顺序）：

```

     ↓
 [7] 用户输入查询
     ↓
 [8] 混合搜索：ES KNN（向量） + BM25（关键词）
     ↓
 [9] Rescore 精排
     ↓
 [10] 结果注入 DeepSeek Prompt
     ↓
 [11] WebSocket 流式返回答案
```

### Q13：文本分块策略？块大小怎么确定的？

**考察点**：Chunking 策略的理解

**回答要点**：

> "我做了可配置的块大小，默认是 512 个字符，重叠 64 个字符。512 这个值不是随便定的——它和 Embedding 模型的最大输入 Token 数、以及 LLM 的上下文窗口做了平衡。块太小会丢失语义完整性，块太大会引入噪声降低检索精度。重叠区是为了避免关键信息刚好被切在边界上。实际测试时对不同文档类型做了对比，技术文档 512-1024 效果最好，合同类文档 256-512 更精确。"

---

### Q14：向量化的 Embedding 模型选型？

**考察点**：Embedding 理解

> "用了阿里通义千问的 Embedding API，输出 2048 维。选阿里的原因是：中文场景效果比同期的开源模型好，而且生产环境 API 稳定性有保障。2048 维是模型的固定输出维度，维度越高表达能力越强但检索计算量也越大。同时我也在代码里预留了 Ollama 本地部署的接口，方便离线场景切换。"

## 2.2 混合搜索篇

### Q15：ES KNN + BM25 双阶段召回怎么实现的？

**考察点**：搜索系统设计（核心亮点的实现细节）

**回答要点**：

```
第一阶段（召回）：
  并行执行两种搜索：
  - ES KNN 向量搜索：查询向量化后，找到语义最相似的 TOP K
  - BM25 关键词搜索：传统的 TF-IDF 加权，找到关键词匹配的 TOP K
  两者结果合并去重

第二阶段（精排）：
  对合并后的结果做 Rescore
  加权公式：score = α * 向量相似度 + (1-α) * BM25 得分
  α 通过实验调参确定（0.6 左右效果最好）
```

**双阶段的意义**：

> "KNN 找的是语义相关的——'买电脑的预算'和'3000-5000 元价位'虽然关键词不同但语义相似。BM25 找的是关键词精确匹配的。两者互补，语义搜索召回率高但可能不精确，关键词搜索精确但查不到同义表达。"

---

### Q16：检索 800ms → 200ms 做了什么优化？

**考察点**：性能优化方法论（最能体现工程能力的点）

**回答框架**（按发现问题→定位→解决→量化）：

| 步骤 | 内容 |
|------|------|
| 发现问题 | 检索平均 800ms，用户感知明显卡顿 |
| 定位瓶颈 | 用日志和链路追踪发现主要耗时在 Embedding 调用（300ms）+ ES 查询（400ms） |
| 优化 1：缓存 | 高频查询的 Embedding 结果缓存到 Redis，相同查询无需重新向量化 |
| 优化 2：ES 索引优化 | 调整分片数、增加字段级 doc_values、优化 mapping |
| 优化 3：并行化 | Embedding 和 ES 查询阶段部分并行 |
| 结果 | 200ms，其中 Redis 命中时 Embedding 耗时降到 5ms |

### Q17：准确率提升 60% 怎么量化的？

## 2.3 架构设计篇

### Q18：为什么用 Kafka 做异步处理？

**考察点**：异步架构设计

> "文档上传后的处理链路很长：解析→分块→向量化→建索引，全部走完可能需要几十秒。如果用同步方式，用户上传完要一直等。Kafka 把处理异步化后，用户上传完立刻得到'上传成功'响应，后台慢慢处理。Kafka 还提供了消息持久化和重试机制，如果向量化失败了会自动重试。"

---

### Q19：分片上传 + 断点续传怎么实现的？

> "前端将大文件切分为多个分片，每个分片独立上传并附带 MD5 校验。后端收到后先校验 MD5 确认完整性，所有分片上传完后触发合并。断点续传的逻辑是：上传前先请求后端查询已上传的分片列表，前端只传缺失的分片。这样网络中断后不需要重传整个文件。"

---

### Q20：多租户权限隔离怎么做的？

> "每个文档关联一个 OrganizationTag，每个用户也关联一个或多个 OrgTag。在 ES 查询时，OrgTagAuthorizationFilter 会自动在查询条件中追加用户的 OrgTag 过滤条件。数据层面天然隔离——不同组织的用户搜索不到对方的文档。同时这个过滤是自动的，业务代码不需要关注权限逻辑。"

---

## 2.4 实时通信篇

### Q21：WebSocket 流式问答怎么实现的？

> "用户在前端发送问题后，后端通过 WebSocket 建立长连接。后端从 DeepSeek 获取流式响应（SSE），每收到一个 token 就通过 WebSocket 推送给前端，前端逐字渲染。同时后端会把完整对话异步持久化到数据库。如果 WebSocket 断了，前端会自动重连，重连后根据会话 ID 恢复上下文。"

# 第三部分：CoderHome — 多模型 AI 社区

## 3.1 多模型集成篇

### Q22：策略模式 + 工厂模式适配多个 LLM 的具体实现？

**考察点**：设计模式应用能力

> "核心接口是 LlmProvider，定义了三个方法：syncCall()、asyncCall()、streamCall()。每个模型提供商实现这个接口（DeepSeekProvider、QwenProvider、SparkProvider 等）。工厂类 LlmProviderFactory 根据配置的 provider 名称返回对应的实例。新增一个模型只需新增一个实现类 + 在工厂里注册，不需要改已有代码。"

**加分点**：主动画类图或伪代码，展示架构能力。

---

### Q23：新增一个模型提供商需要改哪些代码？

> "三步：1) 新建一个类实现 LlmProvider 接口，实现三个调用方法；2) 在工厂类的注册中心加一条映射；3) 在配置文件中添加新模型的配置项。不需要改已有的业务代码，符合开闭原则。"

---

### Q24：同步 / 异步 / 流式怎么统一抽象的？

> "接口定义返回类型时用了回调机制：同步返回 String，异步返回 CompletableFuture<String>，流式返回 void 但通过 Consumer<String> 回调推送给调用方。调用方根据场景选择不同的调用方式——实时聊天用流式，后台批量处理用异步，简单查询用同步。"

## 3.2 自动选路与故障降级篇

### Q25：自动化探测可用模型的具体实现？

> "启动时或定期对所有配置的模型发送一个轻量探测请求（如一段固定文本的 completion），记录响应时间和是否成功。探测结果缓存 10 分钟。用户请求进来时，优先选「可用且响应最快」的模型。如果主模型调用失败（超时/限流/报错），自动切换到下一个可用模型。"

---

### Q26：10 分钟缓存为什么是这个值？过期后探测又失败怎么办？

> "10 分钟是权衡后的选择——太短（1 分钟）会导致频繁探测浪费 API 配额，太长（1 小时）会导致故障发现延迟。如果缓存过期后探测失败，会先重试一次，如果仍然失败就把该模型标记为不可用并延长缓存时间到 30 分钟，避免频繁探测已经挂了的目标。"

---

### Q27：主模型切换到次优模型，进行中的请求怎么处理？

> "当前请求会抛出一个自定义的 ModelUnavailableException，调用方捕获后重新调用降级接口。这是一个优雅降级——不吞异常，不让用户看到空白，而是自动用备用模型重试一次。如果所有模型都挂了，才会返回明确的错误提示。"

---

## 3.3 配置热更新篇

### Q28：MySQL 配置怎么注入 Spring Environment 实时生效？

**考察点**：Spring 源码理解（难点）

> "核心思路是用一个定时任务或事件监听器，定期从 MySQL 读取配置，然后通过 Spring 的 Environment API 更新配置值。对于 @ConfigurationProperties 注解的类，通过反射调用其 setter 方法更新属性值。对于 @Value 注解的字段，通过 Spring 的 ReflectionUtils 更新字段值。整个流程不重启应用。"

**可能的追问**：

- Q：@Value 的修改会并发问题吗？
- A：会的，如果多个线程同时触发热更新。解决方案是加分布式锁或使用 ConcurrentHashMap 的原子操作。

---

### Q29：热更新过程中配置写了一半系统崩了怎么办？

> "我设计了配置版本号机制。每次更新都先写入一条新版本配置，更新完成后切换版本号。如果更新过程崩了，启动时加载的还是上一个完整版本。数据库事务保证了写入的原子性。"

# 第五部分：系统设计开放题

## 5.1 常见开放题

### Q：如果让你设计一个企业级知识库搜索系统，你会怎么做？

**回答框架**：

```
1. 需求分析
   - 上传文档格式（PDF / Word / Excel 等）
   - 搜索方式（关键词 / 语义 / 混合）
   - 权限模型（多租户）
   - 性能要求（检索延迟 / 并发量）

2. 技术选型
   - 存储：MinIO（文件）+ MySQL（元数据）+ ES（搜索）
   - 队列：Kafka（异步处理链路）
   - AI：Embedding API + LLM API

3. 核心流程
   - 上传 → 解析 → 分块 → 向量化 → 建索引
   - 查询 → 混合检索 → 重排 → LLM 生成 → 返回

4. 关键设计
   - 分片上传 + 断点续传
   - 多租户数据隔离
   - 缓存策略
   - 监控与告警

5. 扩展性
   - 支持多种 Embedding 模型切换
   - 支持多种 LLM 切换
   - ES 分片扩展
```

---

### Q：一个 LLM 对话的 API 调用从发送到返回完整经过哪些环节？

```
用户输入
  → 前端发送 HTTP/WebSocket
  → 后端接收
  → RAG 检索（如有）
  → 构造 Prompt（System + History + Context）
  → 调用 LLM API
  → 网络传输到 LLM 服务端
  → LLM 排队等待 GPU 资源
  → LLM 推理生成（主要耗时）
  → 流式返回 Token
  → 后端逐 Token 推给前端
  → 前端逐 Token 渲染

主要延迟分布：
  - 网络延迟（取决于距离和带宽）
  - RAG 检索延迟（如有）
  - LLM 排队延迟（高峰期）
  - LLM 推理延迟（取决于模型大小和 Prompt 长度）
```

# 附录：回答策略速查

## STAR 原则

```
S - Situation：当时的情况是什么
T - Task：需要解决什么问题
A - Action：你具体做了什么
R - Result：取得了什么效果（最好有数据）
```

## 踩坑回答公式

```
坑点：我们最初用了 XXX，结果出现了 YYY 问题
根因：原因是 ZZZ（底层原理）
解决：改成 AAA，因为 BBB（为什么这个方案更好）
反思：如果现在让我重新设计，我会 CCC
```

## 常用过渡语

| 场景 | 话术 |
|------|------|
| 被问到不会的 | "这个细节我目前没有深入研究过，但我的理解是……" |
| 被问到开放题 | "我会先分析需求和约束条件，然后对比几种方案的 trade-off……" |
| 想展示深度 | "这个问题可以从三个层面来看：第一……第二……第三……" |
| 想展示对比 | "我之前试过 XXX 方案，发现 YYY 问题，换成 ZZZ 后解决了" |
